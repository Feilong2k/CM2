# Zettelkasten Knowledge Base: How-To Guide

## Overview

This document describes the system for managing your knowledge across projects (CM2, AGI, etc.) using Zettelkasten principles with Obsidian integration.

## Directory Structure

```
AGI/                          # AGI project root
├── .Docs/                    # Central knowledge base (this folder)
│   ├── 00-INBOX/            # Unprocessed notes awaiting organization
│   ├── ZETTEL/              # Atomic notes (Zettelkasten)
│   └── DOCS/                # Processed historical documents


```

**Key Principle**: `.Docs/` is the **central knowledge base** that can reference files from any project.

## Naming Conventions

### For Zettelkasten Notes

#### Historical Files (extracted from existing documents)
```
YYYYMMDD-idea-slug.md
```
- `YYYYMMDD`: Date the idea **first appeared** (from source file creation/modified date)
- `idea`: Core concept (2-3 words)
- `slug`: URL-friendly version (lowercase, hyphens)

**Example**: `20251229-pcc-router-concept.md` (idea from Dec 29, 2025)

#### New Files (ideas created today)
```
YYYYMMDDHHMM-slug.md
```
- `YYYYMMDDHHMM`: Date and time of note creation
- `slug`: URL-friendly version of the core idea

**Example**: `202601060915-fast-extraction-pattern.md` (created Jan 6, 2026 at 09:15)

### For Source Documents
Keep original filenames in `DOCS/` folder for reference.

## Note Properties (Frontmatter)

Every Zettelkasten note must include this frontmatter:

```yaml
---
aliases: 
  - "Human readable name 1"
  - "Alternative phrasing 2" 
  - "Related term 3"
tags: 
  - project-cm2
  - concept-pcc
  - status-active
  - type-idea
source: "Skills_as_AutoGenerated_Workflows.md#Enhanced Architecture"
source_date: "2026-01-06"
created: "2026-01-06"
updated: "2026-01-06"
---
```

### Property Definitions

| Property | Required | Description | Example |
|----------|----------|-------------|---------|
| `aliases` | **Yes** | 3-5 alternative names for the concept | `["PCC-Router speed", "AGI speed optimization"]` |
| `tags` | **Yes** | Categorization (see Tag Taxonomy below) | `[project-cm2, concept-pcc, status-active]` |
| `source` | For extracted notes | Original document and section | `"Feature2_Skills_v5.md#2-3-11"` |
| `source_date` | For extracted notes | When original was created | `"2025-12-29"` |
| `created` | **Yes** | When this note was created | `"2026-01-06"` |
| `updated` | **Yes** | Last modification date | `"2026-01-06"` |

## Tag Taxonomy

### Project Tags (Always include at least one)
- `project-cm2` - CodeMaestro 2 project
- `project-agi` - AGI project  
- `project-shared` - Concepts applicable to multiple projects

### Concept Tags (Add 1-3 relevant ones)

- `concept-verification` - Verification patterns
- `concept-speed` - Performance optimization
- `concept-architecture` - System architecture
- `concept-llm` - Large Language Models
- `concept-agi` - Artificial General Intelligence

### Status Tags (Always include one)
- `status-inbox` - Needs processing
- `status-active` - Currently being worked on
- `status-reviewed` - Processed and validated
- `status-superseded` - Replaced by newer version of the idea
- `status-historical` - Preserved for historical context, not current practice
- `status-archived` - No longer active (general category)
- `status-reference` - Permanent reference material

### Type Tags (Always include one)
- `type-idea` - Core concept or insight

- `type-worklog` - Daily activity log

## Workflow

### 1. Capture
- **New ideas**: Create note in `ZETTEL/` with current timestamp
- **Existing documents**: Place in `00-INBOX/` for processing
- **Worklogs**: Create in `ZETTEL/` with `type-worklog` tag

### 2. Process
1. For each file in `00-INBOX/`:
   - Extract atomic ideas (one per note)
   - Determine `source_date` from git/file metadata
   - Create Zettelkasten notes in `ZETTEL/`
   - Add proper frontmatter with aliases and tags
   - Move source to `DOCS/` (or delete if no longer needed)

2. Add links between related notes:
   ```markdown
   This builds on [[20251229-pcc-router-concept]].
   Related verification patterns: [[202601060848-rag-verification-example]].
   ```

### 3. Review
- **Daily**: Check Obsidian's "Unlinked Mentions" panel
- **Weekly**: Browse graph view for emerging connections
- **Monthly**: Review and update INDEX files

### 4. Maintain
- Update `updated` date when modifying notes
- Add new aliases as you discover alternative phrasings
- Merge notes when concepts consolidate
- Archive (`status-archived`) obsolete notes

## Linking Strategy

### Explicit Links
- Use `[[note-id]]` syntax to link to other notes
- Link to source documents: `[[source-file#section]]`
- Create bidirectional links where meaningful

### Implicit Links (via Obsidian)
- **Unlinked mentions**: Obsidian detects text matching aliases
- **Graph view**: Visualizes connections between notes
- **Backlinks**: Shows what links to current note

### INDEX Files
Create navigation files for major topics:
```markdown
# INDEX-pcc-router.md

## Core Concepts
- [[20251229-pcc-router-concept]]
- [[202601060847-pcc-router-speed-example]]

## Related Patterns
- [[202601060848-rag-verification-example]]

## Implementation
- [[202601060856-skills-as-workflows-solution-overview]]
```

## Multi-Project Management

### Single Vault Approach
- Open `AGI/.Docs/` as Obsidian vault
- All projects accessible in one interface
- Use `project-*` tags to filter views

### Cross-Project References
- Reference CM2 concepts: `[[project-cm2/note-id]]`
- Use shared tags: `concept-pcc` appears across projects
- Create `SHARED/` folder for cross-cutting concepts

### Project-Specific Views
In Obsidian:
- **Search**: `tag:project-cm2 -tag:project-agi`
- **Graph**: `path:ZETTEL/project-cm2`
- **Tags panel**: Click `project-cm2` to see all CM2 notes

### Note Size
- Notes should be between 200-400 words
- Combine related ideas if less than 200 words, do not combine if total is above 400
- if a unique idea is more than 400 words DO NOT split into multiple files

### Obsidian Plugins to Enable
- **Templater**: For note templates with date variables
- **Dataview**: For querying notes by properties
- **Various Complements**: For alias auto-completion
- **Excalidraw**: For diagrams linked to notes

## Quality Checklist

Before marking a note as `status-reviewed`:

- [ ] Has 3-5 aliases for discoverability
- [ ] Includes all required frontmatter properties
- [ ] Tags follow taxonomy conventions
- [ ] Links to related notes where appropriate
- [ ] Source documented if extracted
- [ ] Written in atomic format (one core idea)
- [ ] No broken links (`[[...]]` references exist)

## Migration from Legacy Systems

### CM2 `.Docs/` to AGI `.Docs/`
1. Process high-value files first (see prioritization below)
2. Extract atomic ideas with proper `source_date`
3. Maintain links to CM2 project files
4. Use `project-cm2` tag for migrated content

### File Processing Priority
1. **Currently active** - Files you've edited recently
2. **Frequently referenced** - Linked from many places
3. **Foundation documents** - Architecture specs, protocols
4. **Historical context** - Older but still relevant

## Troubleshooting

### Unlinked Mentions Not Working
- Ensure aliases are in frontmatter (not just in content)
- Check that alias exactly matches text in other notes
- Restart Obsidian to refresh index

### Graph View Too Crowded
- Use filter: `-tag:status-archived`
- Group by tag: `tag:project-cm2`
- Hide attachments: `-path:assets/`

### Note Discovery Difficult
- Create more INDEX files
- Add more specific aliases
- Use Dataview queries for dynamic indexes
